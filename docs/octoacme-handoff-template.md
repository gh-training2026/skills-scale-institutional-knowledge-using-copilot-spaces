# OctoAcme Hand-off Template

Use this template whenever work moves from one role or team to another. Complete all sections before initiating the hand-off. The receiving role should confirm receipt and raise any questions within one business day.

---

## Hand-off Summary

| Field | Value |
|-------|-------|
| **Hand-off Date** | _YYYY-MM-DD_ |
| **From (Role/Name)** | _e.g., UX Designer / Jane Smith_ |
| **To (Role/Name)** | _e.g., Developer / Alex Lee_ |
| **Project / Feature** | _e.g., User Authentication Redesign_ |
| **Sprint / Milestone** | _e.g., Sprint 12 / Q2 Milestone 2_ |

---

## What Is Being Handed Off

Describe the work artifact(s) being transferred. Include links to relevant files, tickets, or documentation.

- **Artifact(s):** _e.g., Figma design file v3.2, annotated specs for login flow_
- **Location / Link(s):** _e.g., [Figma Link](https://figma.com/...), Ticket #123_
- **Status:** _e.g., Ready for implementation / Needs review / Draft_

---

## Context and Background

Summarize key context the receiving role needs to succeed. Include decisions made, constraints, and relevant history.

_e.g., The design reflects findings from the usability study in Sprint 10. The team agreed to prioritize accessibility compliance (WCAG 2.1 AA). Mobile layout is a P1, desktop is P2._

---

## Acceptance Criteria

List the criteria that define successful completion of this work from the receiving role's perspective.

- [ ] _e.g., Login flow matches design specs in all breakpoints_
- [ ] _e.g., All interactive states are implemented (hover, focus, error)_
- [ ] _e.g., Accessibility requirements are met_

---

## Known Risks or Blockers

List any risks, open questions, or blockers the receiving role should be aware of.

| Risk / Blocker | Owner | Status |
|----------------|-------|--------|
| _e.g., API endpoint not finalized_ | _Backend Lead_ | _Open_ |
| _e.g., Copy for error messages pending_ | _Product Manager_ | _In progress_ |

---

## Dependencies

List any upstream or downstream dependencies that may affect this hand-off.

- **Upstream (needed before starting):** _e.g., Design tokens finalized in design system_
- **Downstream (affected by this work):** _e.g., QA will need updated test cases once implementation is complete_

## Communication During Hand-off

| Question | Raise with | Channel |
|----------|-----------|---------|
| Design clarifications | UX Designer | Slack / comment in Figma |
| Technical blockers | Technical Lead | Standup / Slack |
| Scope or priority changes | Product Manager | Ticket / meeting |
| Schedule or resource issues | Project Manager | Status update / standup |

---

## Checklist for Handoff Giver

- [ ] All artifacts are in agreed locations and access has been confirmed
- [ ] Acceptance criteria are clearly defined
- [ ] Known risks and blockers are documented
- [ ] Receiving role has been notified and hand-off meeting (if needed) is scheduled
- [ ] Relevant tickets are updated with hand-off status

---

## Checklist for Handoff Receiver

- [ ] Reviewed all hand-off artifacts and linked documentation
- [ ] Confirmed understanding of acceptance criteria
- [ ] Identified any immediate blockers or open questions
- [ ] Acknowledged receipt to the handoff giver within one business day
- [ ] Updated ticket status to reflect work is in progress

## Common Hand-off Scenarios

### Design → Development
- UX Designer completes annotated specs and shares with Developer
- Technical Lead confirms technical feasibility
- QA Lead reviews acceptance criteria alignment with test plan

### Development → QA
- Developer marks feature complete and updates ticket
- Provides test environment details and any known edge cases
- QA Lead confirms test environment access and begins test execution

### QA → Release / Project Manager
- QA Lead produces release readiness report
- Documents outstanding defects with severity and owner
- Project Manager reviews go/no-go criteria with stakeholders

### Project Manager → Stakeholder (Milestone Review)
- Project Manager shares status summary and demo plan
- Stakeholder reviews deliverables and provides formal feedback
- Sign-off is documented before release proceeds
